# unit-auth ビジネスロジックモデル

## ビジネスプロセス一覧

| # | プロセス | トリガー | 関連ストーリー |
|---|---|---|---|
| 1 | Googleサインアップ/サインイン | ユーザーが「Googleでログイン」をクリック | US-001 |
| 2 | セッション管理 | 認証済みユーザーのリクエスト | US-001 |
| 3 | サインアウト | ユーザーが「ログアウト」をクリック | US-001 |
| 4 | アカウント削除リクエスト | ユーザーが「アカウント削除」を選択 | US-052 |
| 5 | 削除リクエストキャンセル | 猶予期間中のユーザーがログイン | US-052 |
| 6 | アカウント削除実行 | 猶予期間満了（バッチ処理） | US-052 |

---

## BP-1: Googleサインアップ/サインイン

### フロー

```
1. ユーザーが「Googleでログイン」ボタンをクリック
2. Supabase Auth経由でGoogle OAuthフローを開始
3. Googleの認証・同意画面を表示
4. ユーザーがGoogle側で承認
5. コールバックURLにリダイレクト
6. Supabase Authがトークンを発行
   6a. 新規ユーザー → Userレコード作成（status: ACTIVE）
   6b. 既存ユーザー → last_sign_in_at更新
   6c. PENDING_DELETION状態 → 削除キャンセル確認ダイアログ表示（BP-5へ）
7. アクセストークン（15分）+ リフレッシュトークン（7日）をクライアントに返却
8. ダッシュボードへリダイレクト
```

### 新規ユーザー初期化

- display_name: Googleアカウントの表示名を初期値として設定
- avatar_url: Googleプロフィール画像URLを設定
- status: ACTIVE
- オンボーディングフラグ: true（unit-ui側で初回チュートリアル表示に利用）

## BP-2: セッション管理

### アクセストークンリフレッシュフロー

```
1. クライアントがAPIリクエスト送信
2. ミドルウェアがアクセストークンを検証
   2a. 有効 → リクエスト処理を継続
   2b. 期限切れ → リフレッシュフローへ
3. リフレッシュトークンで新しいアクセストークンを取得
   3a. リフレッシュトークン有効 → 新アクセストークン発行、リクエスト再試行
   3b. リフレッシュトークン期限切れ → 再認証を要求（サインインページへ）
4. 新しいトークンをクライアントに返却
```

### サーバーサイドセッション検証（ミドルウェア）

```
1. Next.js Middlewareでリクエストをインターセプト
2. Supabase Auth Helperでセッション取得
3. セッションなし → 保護ルートの場合はサインインページへリダイレクト
4. セッションあり → ユーザーステータス確認
   4a. ACTIVE → リクエスト処理を継続
   4b. PENDING_DELETION → 削除キャンセル確認ダイアログ表示
   4c. DELETED → セッション破棄、サインインページへリダイレクト
5. ユーザー情報をリクエストコンテキストに注入
```

### ルート保護ルール

| ルートパターン | アクセス制御 |
|---|---|
| `/` (ランディング) | 公開 |
| `/auth/*` | 公開（認証済みの場合はダッシュボードへリダイレクト） |
| `/dashboard/*` | 認証必須 |
| `/ranking` | 認証必須 |
| `/social/*` | 認証必須 |
| `/profile/*` | 認証必須 |
| `/manual/*` | 認証必須 |
| `/api/trpc/*` | 認証必須（tRPCミドルウェアで検証） |

## BP-3: サインアウト

```
1. ユーザーが「ログアウト」をクリック
2. Supabase Auth signOut()を呼び出し
3. 現在のデバイスのセッションのみ破棄
4. クライアント側のキャッシュをクリア
5. サインインページへリダイレクト
```

**Note**: 他デバイスのセッションには影響しない（マルチデバイス制限なし）。

---

## BP-4: アカウント削除リクエスト

```
1. ユーザーがプロフィール設定から「アカウント削除」を選択
2. 確認ダイアログを表示（「本当に削除しますか？」）
3. ユーザーが確認
4. AccountDeletionRequestレコード作成
   - status: PENDING
   - scheduled_deletion_at: 現在時刻 + 7日（猶予期間）
5. Userのstatus → PENDING_DELETION に更新
6. 確認メッセージ表示（「7日後にアカウントが削除されます。期間中にログインするとキャンセルできます。」）
7. サインアウト実行
```

---

## BP-5: 削除リクエストキャンセル（猶予期間中のログイン）

```
1. PENDING_DELETIONステータスのユーザーがログイン
2. サインイン処理は正常に完了
3. 削除キャンセル確認ダイアログを表示
   - 「アカウント削除リクエスト中です。キャンセルしますか？」
   - 選択肢: [キャンセルして利用を続ける] / [削除を継続する]
4a. キャンセル選択:
   - AccountDeletionRequest.status → CANCELLED
   - AccountDeletionRequest.cancelled_at → 現在時刻
   - User.status → ACTIVE
   - ダッシュボードへリダイレクト
4b. 削除継続選択:
   - 状態変更なし
   - ダッシュボードへリダイレクト（PENDING_DELETION状態のまま利用可能）
```

---

## BP-6: アカウント削除実行（バッチ処理）

```
1. 定期バッチ（毎日1回）が実行
2. scheduled_deletion_at < 現在時刻 かつ status = PENDING のレコードを検索
3. 対象ユーザーごとに:
   a. 関連データの論理削除/匿名化
      - display_name → "削除済みユーザー"
      - avatar_url → null
      - email → ハッシュ化（再登録検知用）
   b. 全セッション無効化
   c. User.status → DELETED
   d. AccountDeletionRequest.status → COMPLETED
4. 処理結果をログに記録
```

### 削除時のデータ処理方針

| データ種別 | 処理 | 理由 |
|---|---|---|
| ユーザープロフィール | 匿名化 | ランキング履歴の整合性維持 |
| 認証情報 | 完全削除 | セキュリティ |
| タスク履歴 | 匿名化 | 集計データの整合性維持 |
| AI対話履歴 | 完全削除 | プライバシー |
| ソーシャル関係 | 削除 | 不要 |
| ランキングスコア | 匿名化 | 集計の整合性 |
| 分析データ | 匿名化 | サービス改善用統計 |
